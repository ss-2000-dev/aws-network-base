## 1. 全体像を掴む

AWS のネットワークは「入れ子構造」になっています。 大きなものから順番に理解すると混乱しにくいです。

```
┌─────────────────────────────────────────────────────────┐
│  AWS リージョン（例: ap-northeast-1 東京）               │
│                                                         │
│  ┌───────────────────────────────────────────────────┐  │
│  │  VPC  10.0.0.0/16                                 │  │
│  │                                                   │  │
│  │  ┌─────────────────┐  ┌─────────────────┐        │  │
│  │  │ パブリックサブネット │  │プライベートサブネット│        │  │
│  │  │  10.0.1.0/24    │  │  10.0.2.0/24    │        │  │
│  │  │                 │  │                 │        │  │
│  │  │  ┌───────────┐  │  │  ┌───────────┐  │        │  │
│  │  │  │    EC2    │  │  │  │    RDS    │  │        │  │
│  │  │  │ [SG: web] │  │  │  │ [SG: db]  │  │        │  │
│  │  │  └───────────┘  │  │  └───────────┘  │        │  │
│  │  │                 │  │                 │        │  │
│  │  │  [NACL: public] │  │  [NACL: private]│        │  │
│  │  └─────────────────┘  └─────────────────┘        │  │
│  └───────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
```

**各層の役割まとめ**

|レイヤー|何を守るか|例え|
|---|---|---|
|VPC|AWS 内の独立したネットワーク空間全体|「会社のビル全体」|
|サブネット|VPC を用途ごとに分割したエリア|「ビルの各フロア」|
|NACL|サブネット単位のファイアウォール|「フロアの入口ゲート」|
|セキュリティグループ|インスタンス単位のファイアウォール|「部屋の鍵」|

## 2. VPC

### VPC とは

VPC（Virtual Private Cloud）は、AWS 上に作る「自分専用のプライベートネットワーク空間」です。 インターネットから論理的に隔離されており、VPC の中に EC2 や RDS などのリソースを配置します。

デフォルトでは、異なる AWS アカウントの VPC 同士は通信できません。

### VPC を作る際に決めること

|設定項目|説明|例|
|---|---|---|
|CIDR ブロック|VPC が使える IP アドレスの範囲|`10.0.0.0/16`|
|リージョン|VPC を作成するリージョン|`ap-northeast-1`（東京）|
|テナント|ハードウェア共有 or 専有|通常は `default`（共有）|

### CIDR ブロックの選び方

VPC の CIDR は `/16`〜`/28` の範囲で指定します。  
プライベートアドレス範囲（RFC 1918）を使うのが一般的です。

|プライベートアドレス範囲|利用可能な IP 数|使い方|
|---|---|---|
|`10.0.0.0/8`|約 1,677 万|大規模システム向け|
|`172.16.0.0/12`|約 104 万|中規模向け|
|`192.168.0.0/16`|約 6.5 万|小規模・PoC 向け|

> **推奨**: `10.0.0.0/16` を VPC に割り当てるのが一般的。  
> 将来の VPC Peering や Transit Gateway 接続を考え、他の VPC と CIDR が重複しないよう設計する。

### VPC の主要コンポーネント

```
VPC
 ├── インターネットゲートウェイ (IGW)     ... インターネットへの出入口
 ├── NAT ゲートウェイ                    ... プライベートサブネットからの外向き通信
 ├── ルートテーブル                      ... パケットの経路を定義
 ├── サブネット                          ... IP 範囲のグループ
 ├── セキュリティグループ (SG)           ... インスタンス単位のファイアウォール
 └── ネットワーク ACL (NACL)            ... サブネット単位のファイアウォール
```

## 3. サブネットと CIDR

### サブネットとは

サブネットは VPC の IP アドレス空間を分割した「区画」です。  
サブネットは必ず 1 つのアベイラビリティゾーン（AZ）に属します。

**パブリックサブネット vs プライベートサブネット**

|種類|インターネット直接通信|用途例|
|---|---|---|
|パブリックサブネット|✅ IGW 経由で可能|Web サーバー、ALB、踏み台サーバー|
|プライベートサブネット|❌ NAT GW 経由のみ（アウトバウンドのみ）|アプリサーバー、DB、バックエンド処理|

> **パブリックかどうかを決めるのは「ルートテーブルに IGW へのルートがあるかどうか」です。**  
> サブネット名は関係ありません。

### CIDR とは

CIDR（Classless Inter-Domain Routing）は IP アドレスの範囲を表す記法です。  
`IPアドレス / プレフィックス長` の形式で書きます。

```
10.0.1.0/24
│         │
│         └─ プレフィックス長（ネットワーク部のビット数）
└─────────── ネットワークアドレス
```

**プレフィックス長と利用可能な IP 数の関係**

AWS では各サブネットの先頭 4 つと末尾 1 つ（合計 5 つ）の IP は AWS が予約します。

|CIDR|総 IP 数|AWS 予約|利用可能 IP 数|目安|
|---|---|---|---|---|
|`/28`|16|5|**11**|極小・テスト用|
|`/27`|32|5|**27**|小規模|
|`/26`|64|5|**59**|小〜中規模|
|`/25`|128|5|**123**|中規模|
|`/24`|256|5|**251**|一般的な選択|
|`/23`|512|5|**507**|中〜大規模|
|`/22`|1,024|5|**1,019**|大規模|

### サブネット分割の実例

VPC `10.0.0.0/16` を以下のように分割するのが典型的なパターンです。

```
VPC: 10.0.0.0/16（65,536 IP）
│
├── AZ-a（ap-northeast-1a）
│   ├── パブリックサブネット  : 10.0.1.0/24（251 IP）
│   ├── プライベートサブネット: 10.0.11.0/24（251 IP）
│   └── DB サブネット        : 10.0.21.0/24（251 IP）
│
└── AZ-c（ap-northeast-1c）
    ├── パブリックサブネット  : 10.0.2.0/24（251 IP）
    ├── プライベートサブネット: 10.0.12.0/24（251 IP）
    └── DB サブネット        : 10.0.22.0/24（251 IP）
```

**サブネット分割のベストプラクティス**

- 用途ごとにサブネットを分割する（Web / App / DB）
- 複数 AZ にまたがって配置し、高可用性を確保する
- サブネットの CIDR は後から変更できないため、最初に余裕を持って設計する
- 将来の拡張を見越して VPC の CIDR も大きめに取る（`/16` 推奨）

### AWS 予約 IP の内訳（例: `10.0.1.0/24`）

|IP アドレス|AWS の用途|
|---|---|
|`10.0.1.0`|ネットワークアドレス（使用不可）|
|`10.0.1.1`|VPC ルーター|
|`10.0.1.2`|DNS サーバー|
|`10.0.1.3`|将来利用のための予約|
|`10.0.1.255`|ブロードキャストアドレス（使用不可）|

## 4. セキュリティグループ

### セキュリティグループとは

セキュリティグループ（SG）は **EC2 などのインスタンス（ENI）に紐づく仮想ファイアウォール** です。  
「どこからの・どのポートへの・通信を許可するか」をルールで定義します。

**重要な特徴**

- **ステートフル**：インバウンドを許可したら、そのレスポンスのアウトバウンドは自動的に許可される
- **許可ルールしか書けない**（明示的な拒否はできない）
- 1 つのインスタンスに複数の SG を割り当てられる
- SG 同士を参照して「同じ SG を持つインスタンスからの通信を許可」という書き方ができる

### ルールの構成要素

|項目|説明|例|
|---|---|---|
|タイプ|通信の種類|HTTP, HTTPS, SSH, カスタム TCP など|
|プロトコル|TCP / UDP / ICMP|TCP|
|ポート範囲|対象ポート|`80`, `443`, `3306`, `0-65535`|
|ソース（インバウンド）|許可する通信元|`0.0.0.0/0`（全許可）, `10.0.0.0/16`, SG ID|
|送信先（アウトバウンド）|許可する通信先|`0.0.0.0/0`, IP 範囲, SG ID|

### SG 参照（SG to SG）の活用

IP アドレスではなく、別のセキュリティグループ ID を参照できます。  
これにより「ALB からのリクエストのみ App サーバーへ届ける」という制御が実現できます。

```
[ユーザー] → [ALB: sg-alb] → [EC2: sg-app]
                                      ↑
                    インバウンドルールに sg-alb を指定
                    「sg-alb を持つリソースからの 8080 番を許可」
```

### セキュリティグループの設計例（3層構成）

#### SG-web（ALB 用）

|方向|プロトコル|ポート|送受信先|目的|
|---|---|---|---|---|
|インバウンド|TCP|80|`0.0.0.0/0`|HTTP を全許可|
|インバウンド|TCP|443|`0.0.0.0/0`|HTTPS を全許可|
|アウトバウンド|TCP|8080|`sg-app`|App サーバーへ転送|

#### SG-app（EC2 アプリサーバー用）

|方向|プロトコル|ポート|送受信先|目的|
|---|---|---|---|---|
|インバウンド|TCP|8080|`sg-web`|ALB からのみ許可|
|インバウンド|TCP|22|`sg-bastion`|踏み台からの SSH のみ|
|アウトバウンド|TCP|3306|`sg-db`|DB への接続|
|アウトバウンド|TCP|443|`0.0.0.0/0`|HTTPS 外部 API 呼び出し|

#### SG-db（RDS 用）

|方向|プロトコル|ポート|送受信先|目的|
|---|---|---|---|---|
|インバウンド|TCP|3306|`sg-app`|App サーバーからのみ許可|
|アウトバウンド|─|─|─|（不要なため全削除）|

## 5. ネットワーク ACL

### ネットワーク ACL とは

ネットワーク ACL（NACL: Network Access Control List）は **サブネット単位** に設定するファイアウォールです。  
サブネットを出入りするすべてのトラフィックに対して評価されます。

**重要な特徴**

- **ステートレス**：インバウンドとアウトバウンドは独立して評価される（レスポンスも明示的に許可が必要）
- **許可と拒否の両方**のルールを書ける
- ルール番号が小さい順に評価され、**最初にマッチしたルールが適用される**
- 1 つのサブネットに 1 つの NACL のみ関連付けられる
- デフォルト NACL はすべての通信を許可する

### ルール番号の考え方

```
ルール 100 → ルール 200 → ルール 300 → ・・・ → * DENY（デフォルト）
         ↑最初にマッチしたルールで判定終了
```

> **推奨**: ルール番号は 100 刻みや 10 刻みで作成し、後から挿入できる余白を作っておく。

### NACL の設計例（パブリックサブネット用）

#### インバウンドルール

|ルール番号|プロトコル|ポート|送信元|アクション|目的|
|---|---|---|---|---|---|
|100|TCP|80|`0.0.0.0/0`|ALLOW|HTTP|
|110|TCP|443|`0.0.0.0/0`|ALLOW|HTTPS|
|120|TCP|22|`203.0.113.0/24`（自社 IP）|ALLOW|SSH（自社のみ）|
|130|TCP|1024-65535|`0.0.0.0/0`|ALLOW|レスポンスの戻りパケット（エフェメラルポート）|
|*|ALL|ALL|`0.0.0.0/0`|DENY|その他すべて拒否|

#### アウトバウンドルール

|ルール番号|プロトコル|ポート|送信先|アクション|目的|
|---|---|---|---|---|---|
|100|TCP|80|`0.0.0.0/0`|ALLOW|HTTP アウトバウンド|
|110|TCP|443|`0.0.0.0/0`|ALLOW|HTTPS アウトバウンド|
|120|TCP|1024-65535|`0.0.0.0/0`|ALLOW|クライアントへのレスポンス（エフェメラルポート）|
|*|ALL|ALL|`0.0.0.0/0`|DENY|その他すべて拒否|

## 6. SG vs NACL：何が違うのか

### 主要な違い一覧

|比較項目|セキュリティグループ（SG）|ネットワーク ACL（NACL）|
|---|---|---|
|**適用対象**|インスタンス（ENI）単位|サブネット単位|
|**ステートフル/レス**|**ステートフル**（レスポンスは自動許可）|**ステートレス**（インバウンド・アウトバウンドを別々に設定）|
|**ルールの種類**|許可ルールのみ|許可と拒否の両方|
|**ルール評価**|すべてのルールを評価し、1 つでも許可があれば許可|ルール番号順に評価し、最初にマッチしたルールを適用|
|**デフォルト動作**|すべて拒否（インバウンド）、すべて許可（アウトバウンド）|すべて許可（デフォルト NACL）|
|**変更の反映**|即時|即時|
|**1 リソースへの紐付け数**|最大 5 個（デフォルト。上限緩和可能）|1 つのみ|
|**エフェメラルポート**|不要（ステートフルなので自動処理）|必要（ステートレスなので明示的に許可）|

### ステートフル vs ステートレスの具体的な違い

#### セキュリティグループ（ステートフル）の動き

```
クライアント（1.2.3.4）からポート 443 で接続

[インバウンド評価]
  ルール: TCP 443 を 0.0.0.0/0 から許可  → ✅ 許可

[アウトバウンド評価（レスポンス）]
  ※ステートフルなので、このレスポンスは自動的に追跡・許可される
  → アウトバウンドルールを別途書く必要なし
```

#### ネットワーク ACL（ステートレス）の動き

```
クライアント（1.2.3.4）からポート 443 で接続
クライアントのエフェメラルポートは 54321 番を使用

[インバウンド評価]
  ルール: TCP 443 を 0.0.0.0/0 から許可  → ✅ 許可

[アウトバウンド評価（レスポンス）]
  ※ステートレスなので、レスポンスも独立してルール評価される
  レスポンスは 1.2.3.4:54321（エフェメラルポート）へ送信
  ルール: TCP 1024-65535 を 0.0.0.0/0 へ許可  → ✅ 許可（このルールが必要！）
  ルールがなければ → ❌ DENY → 接続が切れる
```

### エフェメラルポートとは

クライアントが通信を開始するとき、OS がランダムに選ぶ送信元ポートのことです。  
TCP の場合、通常は `1024〜65535` の範囲から選ばれます。

```
クライアント（送信元: 54321）  →  サーバー（送信先: 443）
クライアント（受信: 54321）    ←  サーバー（送信元: 443）

エフェメラルポート = 54321
（OS が自動で選ぶ一時的なポート番号）
```

NACL はステートレスなので、このレスポンス（54321 番へのアウトバウンド）を  
**明示的に許可する必要があります**。

> **推奨**: NACL のアウトバウンドルールには `TCP 1024-65535` を許可するルールを入れておく。  
> Linux のエフェメラルポート範囲: `32768-60999`  
> AWS / Windows: `1024-65535`（広めに設定するのが安全）

## 7. リクエストの流れを完全に追う

### シナリオ設定

```
構成:
  ユーザー（Internet）
    → ALB（パブリックサブネット 10.0.1.0/24）
      → EC2 アプリサーバー（プライベートサブネット 10.0.11.0/24）
        → RDS（DB サブネット 10.0.21.0/24）

セキュリティグループ:
  sg-alb   : ALB 用
  sg-app   : EC2 用
  sg-db    : RDS 用

NACL:
  nacl-public  : パブリックサブネット（10.0.1.0/24）
  nacl-private : プライベートサブネット（10.0.11.0/24）
  nacl-db      : DB サブネット（10.0.21.0/24）
```

---

### フェーズ 1：ユーザー → ALB（HTTPS リクエスト）

```
ユーザー IP: 1.2.3.4
  接続先: ALB（10.0.1.10:443）
  エフェメラルポート: 51000
```

#### Step 1-1：nacl-public（パブリックサブネット）のインバウンド評価

NACL はサブネットの**入口**で評価されます。

|ルール|プロトコル|ポート|送信元|アクション|判定|
|---|---|---|---|---|---|
|100|TCP|443|`0.0.0.0/0`|ALLOW|✅ マッチ → ALLOW で確定|

→ パケットがサブネット内に入ることを許可

#### Step 1-2：sg-alb のインバウンド評価

SG はサブネット通過後、インスタンス（ENI）に到達する際に評価されます。

|ルール|プロトコル|ポート|送信元|判定|
|---|---|---|---|---|
|ALLOW TCP 443|TCP|443|`0.0.0.0/0`|✅ 許可|

→ ALB がリクエストを受け付ける

---

### フェーズ 2：ALB → EC2（HTTP 転送）

ALB がリクエストを EC2（プライベートサブネット）に転送します。

```
ALB IP: 10.0.1.10
  接続先: EC2（10.0.11.20:8080）
  エフェメラルポート: 62000
```

#### Step 2-1：nacl-public（パブリックサブネット）のアウトバウンド評価

ALB がパブリックサブネットから出るときに評価されます。

|ルール|プロトコル|ポート|送信先|アクション|判定|
|---|---|---|---|---|---|
|100|TCP|1024-65535|`0.0.0.0/0`|ALLOW|✅ ポート8080はこの範囲 → ALLOW|

→ パケットがサブネットを出ることを許可

#### Step 2-2：nacl-private（プライベートサブネット）のインバウンド評価

EC2 があるプライベートサブネットの入口で評価されます。

|ルール|プロトコル|ポート|送信元|アクション|判定|
|---|---|---|---|---|---|
|100|TCP|8080|`10.0.1.0/24`|ALLOW|✅ マッチ → ALLOW|

→ ALB からのパケットがプライベートサブネットに入ることを許可

#### Step 2-3：sg-app のインバウンド評価

|ルール|プロトコル|ポート|送信元|判定|
|---|---|---|---|---|
|ALLOW TCP 8080|TCP|8080|`sg-alb`（SG 参照）|✅ ALB の SG が一致 → 許可|

→ EC2 がリクエストを受け付ける

---

### フェーズ 3：EC2 → RDS（DB 接続）

EC2 がアプリ処理中に RDS に接続します。

```
EC2 IP: 10.0.11.20
  接続先: RDS（10.0.21.30:3306）
  エフェメラルポート: 55000
```

#### Step 3-1：sg-app のアウトバウンド評価

|ルール|プロトコル|ポート|送信先|判定|
|---|---|---|---|---|
|ALLOW TCP 3306|TCP|3306|`sg-db`（SG 参照）|✅ 許可|

※ SG はステートフルなので、RDS からのレスポンスも自動で許可される

#### Step 3-2：nacl-private（プライベートサブネット）のアウトバウンド評価

|ルール|プロトコル|ポート|送信先|アクション|判定|
|---|---|---|---|---|---|
|100|TCP|3306|`10.0.21.0/24`|ALLOW|✅ 許可|

#### Step 3-3：nacl-db（DB サブネット）のインバウンド評価

|ルール|プロトコル|ポート|送信元|アクション|判定|
|---|---|---|---|---|---|
|100|TCP|3306|`10.0.11.0/24`|ALLOW|✅ 許可|

#### Step 3-4：sg-db のインバウンド評価

|ルール|プロトコル|ポート|送信元|判定|
|---|---|---|---|---|
|ALLOW TCP 3306|TCP|3306|`sg-app`（SG 参照）|✅ 許可|

→ RDS が EC2 からの接続を受け付ける

### フェーズ 4：レスポンスの戻り

EC2 からのレスポンスがユーザーに返るまでの逆方向の評価。

```
RDS → EC2 のレスポンス:
  RDS: sg-db（ステートフル → 自動許可）
  nacl-db アウトバウンド: TCP 1024-65535 → 10.0.11.0/24 を ALLOW（要明示）
  nacl-private インバウンド: TCP 1024-65535 → ALLOW（要明示）
  EC2: sg-app（ステートフル → 自動許可）

EC2 → ALB のレスポンス:
  sg-app（ステートフル → 自動許可）
  nacl-private アウトバウンド: TCP 1024-65535 を ALLOW（要明示）
  nacl-public インバウンド: TCP 1024-65535 を ALLOW（要明示）
  sg-alb（ステートフル → 自動許可）

ALB → ユーザーのレスポンス:
  sg-alb（ステートフル → 自動許可）
  nacl-public アウトバウンド: TCP 1024-65535（エフェメラル）を ALLOW（要明示）
```

> **まとめ**: SG は戻りパケットを意識しなくてよいが、  
> **NACL は戻りパケット用にエフェメラルポート範囲（1024-65535）を双方向で許可する必要がある。**

## 9. よくあるハマりポイント

### ❌ ハマり 1：NACL でエフェメラルポートを忘れる

**症状**: インバウンドは通るのにレスポンスが返ってこない、接続がタイムアウトする

**原因**: NACL のアウトバウンドに TCP 1024-65535 の許可ルールを追加していない

**解決策**:

```
アウトバウンド: TCP 1024-65535 0.0.0.0/0 → ALLOW を追加
インバウンド:  TCP 1024-65535 0.0.0.0/0 → ALLOW を追加（双方向で必要）
```

---

### ❌ ハマり 2：SG の「すべてのトラフィックを許可」アウトバウンドを削除する

**症状**: アプリが外部 API に接続できない、S3 にアクセスできない

**原因**: SG のデフォルトのアウトバウンドルール（全許可）を誤って削除してしまった

**解決策**: 必要な送信先を明示的に許可するか、デフォルトルールを復元する

---

### ❌ ハマり 3：NACL のルール番号の順序を間違える

**症状**: 特定の IP を許可したつもりが通らない、または拒否したつもりが通る

**原因**: NACL は番号順に評価するため、広い許可ルール（例: 100 で 0.0.0.0/0 を ALLOW）が  
狭い拒否ルール（例: 200 で特定 IP を DENY）より前にある

**解決策**: 拒否ルールを許可ルールより小さい番号に配置する

```
# 正しい順序（特定 IP を先に DENY）
ルール 90 : TCP 443 1.2.3.4/32 → DENY（特定 IP のみブロック）
ルール 100: TCP 443 0.0.0.0/0  → ALLOW（その他は許可）
```

---

### ❌ ハマり 4：SG の参照が正しく機能しない

**症状**: sg-app から sg-alb を参照したが、ALB からの通信が届かない

**原因**: ALB に sg-alb が紐付いていない、または別のリージョン・VPC の SG を参照している

**確認手順**:

1. ALB に `sg-alb` が割り当てられているか確認
2. SG が同じ VPC 内のものか確認
3. 送受信の方向（インバウンド/アウトバウンド）が正しいか確認

---

### ❌ ハマり 5：サブネット CIDR の重複

**症状**: VPC Peering や Transit Gateway 接続ができない

**原因**: 接続先の VPC のサブネット CIDR と自 VPC のサブネット CIDR が重複している

**解決策**: VPC 設計時に将来の接続先も見越して CIDR を重複しないよう管理する

```
✅ 良い例:
  VPC-A: 10.0.0.0/16
  VPC-B: 10.1.0.0/16
  VPC-C: 10.2.0.0/16

❌ 悪い例:
  VPC-A: 10.0.0.0/16
  VPC-B: 10.0.0.0/16  ← 重複！Peering 不可
```

---

## 付録：チェックリスト

### サブネット設計チェック

- [ ] VPC の CIDR は十分に大きいか（推奨 `/16`）
- [ ] 複数 AZ にサブネットを分散しているか
- [ ] パブリック / プライベート / DB の 3 層に分けているか
- [ ] 将来の拡張に備えて CIDR に余裕があるか
- [ ] 他 VPC との CIDR 重複がないか

### セキュリティグループチェック

- [ ] 最小権限の原則に従っているか（不要なポートは閉じているか）
- [ ] `0.0.0.0/0` の許可は本当に必要な場合のみか
- [ ] SG 参照を活用して IP 指定を減らしているか
- [ ] アウトバウンドルールで不要な通信を制限しているか

### NACL チェック

- [ ] エフェメラルポート（1024-65535）をインバウンド・アウトバウンド両方で許可しているか
- [ ] ルール番号に余白（100 刻みなど）があるか
- [ ] 拒否ルールが許可ルールより前の番号になっているか
- [ ] 各サブネットに適切な NACL が関連付けられているか
