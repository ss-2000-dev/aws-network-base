「インターネットからアクセスされる制限やポリシーが管理されていること」という要件は、AWS では複数のレイヤーにまたがります。整理してお伝えします。

---
## 要件の解釈

この要件は大きく2つの観点に分解できます。

- **インバウンド制限**：インターネットからAWSリソースへのアクセスを制御する
- **ポリシー管理**：誰が・何に・どうアクセスできるかをポリシーで明文化・管理する

---
## 該当するAWSリソース・設定

### 🔒 ネットワーク層（インバウンド制限）

| リソース                     | 役割                             | ポイント                          |
| ------------------------ | ------------------------------ | ----------------------------- |
| **VPC**                  | ネットワーク境界の基本単位                  | パブリック/プライベートサブネットの分離設計が前提     |
| **セキュリティグループ**           | インスタンス単位のステートフルなファイアウォール       | 許可ルールのみ定義。不要ポートを開けない          |
| **ネットワーク ACL（NACL）**     | サブネット単位のステートレスなファイアウォール        | セキュリティグループの外側の防御層             |
| **インターネットゲートウェイ（IGW）**   | VPCとインターネットの接続点                | プライベートサブネットには関連付けない           |
| **NAT ゲートウェイ**           | プライベートサブネットからの一方向アウトバウンドのみ許可   | インバウンドは遮断される                  |
| **AWS WAF**              | L7（アプリ層）のアクセス制御                | 先ほどのログフィールドが活きるリソース           |
| **AWS Shield**           | DDoS 攻撃からの保護                   | Standard は自動適用、Advanced は追加設定 |
| **AWS Network Firewall** | VPC 内のトラフィックを検査するマネージドファイアウォール | より細かいルールが必要な場合に使用             |

---
### 🧑‍💼 アクセス管理・ポリシー層

| リソース | 役割 | ポイント |
|---|---|---|
| **IAM ポリシー** | AWS リソースへの操作権限を定義 | 最小権限の原則（Least Privilege）が基本 |
| **IAM ロール** | サービス・ユーザーに権限を付与する仕組み | アクセスキー直書きを避けるために使用 |
| **S3 バケットポリシー** | S3 への公開・非公開を制御 | パブリックアクセスブロックが有効か確認 |
| **S3 パブリックアクセスブロック** | バケット・アカウント単位で公開設定を強制 | 商用環境では基本的にすべて有効化 |
| **リソースベースポリシー** | S3/SQS/SNS 等のリソース側に付与するポリシー | クロスアカウントアクセスの制御にも使用 |
| **AWS Organizations SCP** | アカウントやOUに対してポリシーを強制適用 | 組織全体のガードレール設定に使用 |

---
### 🔍 検出・可視化・監査層（ポリシーが守られているかの確認）

| リソース | 役割 |
|---|---|
| **AWS Config** | リソース設定の変更を記録・コンプライアンス評価 |
| **AWS CloudTrail** | API 操作ログ（誰が・何を・いつ操作したか） |
| **Amazon GuardDuty** | 不審なアクセスや脅威をAIで自動検出 |
| **AWS Security Hub** | セキュリティチェックの統合ダッシュボード |
| **Amazon Inspector** | EC2/コンテナの脆弱性スキャン |
| **AWS Trusted Advisor** | セキュリティのベストプラクティス逸脱を自動チェック |

## 設計の考え方（レイヤー構造）

```
インターネット
     │
  [Route 53 / CloudFront]  ← エッジでの制御
     │
  [AWS WAF / Shield]        ← L7 アクセス制御・DDoS 防御
     │
  [ALB / API Gateway]       ← エンドポイント（パブリックサブネット）
     │
  [セキュリティグループ / NACL] ← ネットワーク層フィルタ
     │
  [EC2 / ECS / Lambda]      ← プライベートサブネット
     │
  [RDS / S3 / DynamoDB]     ← データ層（直接インターネット非公開）
```

この多層防御の構造全体が「インターネットからのアクセスを制限・管理している」という要件の回答になります。

---
## よくある審査・認定との対応

もし要件が特定のフレームワーク（ISMS、SOC2、PCI DSS など）に基づいている場合、AWS 側では **AWS Artifact** でコンプライアンスレポートを入手でき、自社の統制証跡として活用できます。
要件の出どころ（社内規程なのか・外部認証なのか）がわかれば、どのリソースを優先すべきかもう少し絞り込めます。

---
### AWS Network Firewall

良い指摘です。AWS Network Firewall はこの位置に入ります。

```
インターネット
     │
  [Route 53 / CloudFront]        ← エッジでの制御
     │
  [AWS WAF / Shield]              ← L7 アクセス制御・DDoS 防御
     │
  [ALB / API Gateway]             ← エンドポイント（パブリックサブネット）
     │
  [AWS Network Firewall]          ← ★ここ（L3〜L7 のステートフル/ステートレス検査）
     │
  [セキュリティグループ / NACL]    ← ネットワーク層フィルタ
     │
  [EC2 / ECS / Lambda]            ← プライベートサブネット
     │
  [RDS / S3 / DynamoDB]           ← データ層（直接インターネット非公開）
```

---

### WAF との違いと使い分け

| 比較軸        | AWS WAF                     | AWS Network Firewall         |
| ---------- | --------------------------- | ---------------------------- |
| 対象レイヤー     | L7（HTTP/HTTPS）              | L3〜L7（IP/ポート〜アプリ層）           |
| 主な用途       | Web アプリへの攻撃防御（SQLi, XSS など） | VPC 内トラフィック全体の検査・制御          |
| 通信方向       | インバウンドのみ                    | インバウンド・アウトバウンド・East-West 両対応 |
| ドメインフィルタ   | ✗                           | ✅（例：特定ドメインへのアウトバウンドのみ許可）     |
| IDS/IPS 機能 | ✗                           | ✅（Suricata ルール互換）            |

実際の商用環境では **WAF と Network Firewall を併用**するケースが多く、「Web 攻撃は WAF で、VPC 内の通信制御・アウトバウンド制限は Network Firewall で」と役割分担するのが一般的です。
