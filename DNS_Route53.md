### 名前解決の流れ
```
ユーザーが lp.aichi.nihon.com にアクセスしたいとき

[クライアント]
   ↓ ① 問い合わせ
[フルリゾルバ (キャッシュDNSサーバ)]
   ↓ ② キャッシュになければ、ルートから順にたどる
[ルートDNSサーバ]        →「.com は あっちのサーバに聞け」
   ↓
[.com の権威サーバ (TLD)] →「nihon.com は あっちに聞け」
   ↓
[nihon.com の権威サーバ]  →「lp.aichi... のレコードはこれだ」
   ↓ ③ IP アドレスを返す
[クライアント] → 取得した IP に接続
```

----
### 権威サーバ (Authoritative Name Server)

あるドメインについて「正しい答えを持っている責任者」
Route 53 のホストゾーンはこの権威サーバの役割を担う

#### 例
`example.com` を会社として持っていて、東京支社に `tokyo.example.com`、その配下にウェブサーバ `web.tokyo.example.com` を置く

`tokyo.example.com` をホストゾーン（＝ゾーン）として作った場合、その権威サーバは `tokyo.example.com` **以下**の名前について正しく答える責任を持つ。

```
tokyo.example.com           ← 答えられる
web.tokyo.example.com       ← 答えられる
db.tokyo.example.com        ← 答えられる
api.web.tokyo.example.com   ← 答えられる（さらに下も含む）

example.com                 ← 答えられない（これは親ゾーンの責任範囲）
osaka.example.com           ← 答えられない（別の責任範囲）
```

ポイントは「責任範囲はゾーン単位で区切られる」ことです。`example.com` の権威サーバと `tokyo.example.com` の権威サーバは**別々に存在しうる**（同じでもよい）。その区切りを作るのが後述の **委任** です。

### ゾーンファイル

ゾーンファイルは、**1 つのゾーンに属するすべての DNS レコードを記述したテキストファイル**です。Route 53 のホストゾーンは、このゾーンファイルを GUI / API でラップしたもの

典型的な `tokyo.example.com` のゾーンファイルの例

```dns
$ORIGIN tokyo.example.com.
$TTL 3600

@       IN  SOA  ns-1.awsdns-01.com. admin.example.com. (
                    2024010101  ; Serial（シリアル番号）
                    7200        ; Refresh
                    900         ; Retry
                    1209600     ; Expire
                    86400 )     ; Minimum TTL

@       IN  NS   ns-1.awsdns-01.com.
@       IN  NS   ns-2.awsdns-02.net.

web     IN  A    192.0.2.10
db      IN  A    192.0.2.20
@       IN  A    192.0.2.1
```

読み解きの鍵：
- **`$ORIGIN`**：このゾーンの基準名。以降の相対表記はこれを補完する。
- **`@`** ：`$ORIGIN` そのもの（ここでは `tokyo.example.com.`）を指す記号。
- **末尾ドットの有無**：
    - `web` → ドットなし＝相対表記 → `web` + `$ORIGIN` = `web.tokyo.example.com.`
    - `ns-1.awsdns-01.com.` → ドットあり＝FQDN＝そのまま確定（`$ORIGIN` を付けない）
    - もし `ns-1.awsdns-01.com`（ドット忘れ）と書くと `ns-1.awsdns-01.com.tokyo.example.com.` という誤った名前になる。これが古典的なバグです。
- 各行 `名前 IN レコード種別 値`。`IN` は Internet クラスの意味（ほぼ常にこれ）。

##### SOA レコード
SOA（Start Of Authority）は、**ゾーンの管理情報を1つだけ持つレコード**で、ゾーンを作ると必ず先頭に自動生成されます。「このゾーンはここから権威が始まる」という宣言です。

##### NSレコード
NS（Name Server）レコードは、「この名前の権威サーバはこれだ」と示すレコードNS には 2 つの登場場面があり、ここが委任の核心です。

```
① ゾーン自身の頂点に置く NS（自分の権威サーバ宣言）
   tokyo.example.com.  →  ns-1.awsdns-01.com.  （tokyo ゾーン内）

② 親ゾーンに置く NS（子への委任）
   tokyo.example.com.  →  ns-1.awsdns-01.com.  （example.com ゾーン内）
```

同じ内容でも**置かれるゾーンが違う**ことに注目してください。①は子ゾーン自身が「私の権威サーバはこれ」と言っているだけ。②は親が「`tokyo` のことはこの NS に聞け」と外に向かって案内している。**外部のリゾルバが実際にたどるのは②**です。

##### NSレコードで指しているNSサーバが実際にしていること

NS レコードが指しているのは IP アドレスではなく、**サーバの名前（ドメイン名）

```
NS  tokyo.example.com.  →  ns-1.awsdns-01.com.   ← サーバの名前であって IP ではない
```

つまり「`tokyo.example.com` のことは、`ns-1.awsdns-01.com.` という名前のサーバに聞け」と言っているだけ。実際に通信するには、この `ns-1.awsdns-01.com.` を**さらに A レコードで IP に解決**する必要がある。

```
ns-1.awsdns-01.com.  →  A  205.251.192.1   ← 最終的にはこの IP のマシン
```

----
#### ネームサーバが実際にしていること

ネームサーバ（権威サーバ）は、**「自分が責任を持つゾーンのレコードを保持し、問い合わせに答えるサーバ」**です。具体的な仕事は2種類です。

##### ① 答えを返す（自分が権威を持つ名前の場合）

```
リゾルバ: 「web.tokyo.example.com の A は？」
   ▼
ns-1 (tokyo.example.com の権威):
   ゾーンファイルを検索 → 「192.0.2.10 です」(Authoritative Answer)
```

自分のゾーンに該当レコードがあれば、**確定的な答え（権威ある回答）**を返します。

##### ② 委任先を案内する（自分の子ゾーンの場合）

```
リゾルバ: 「example.com の権威サーバが osaka.example.com を聞かれた」
   ▼
example.com の権威サーバ:
   自分のゾーンに osaka の NS がある → 「osaka のことは ns-9.awsdns-xx に聞いて」
   (Referral = 紹介)
```

自分では答えず、**「次はここに聞け」と案内（リファラル）**します。これが委任の実行部分

重要な性質として、**権威サーバは原則「再帰しない」**点があり、リゾルバ（フルサービスリゾルバ）はルートから順にたどって最終回答まで責任を持つが、権威サーバは「自分の担当範囲を答える / 委任先を紹介する」だけで、代わりにたどってあげることはしません。役割が明確に分かれています。

### Route 53 ホストゾーン

あるドメイン（およびそのサブドメイン）に関する DNS レコードをまとめて管理するコンテナ
DNS の用語でいう「ゾーンファイル」に相当し、Route 53 ではこのホストゾーンが権威サーバとして機能している

ホストゾーンを作成すると、Route 53 は自動的に以下を生成する
- **NS レコード**：このゾーンを管理する 4 つの Route 53 ネームサーバ
- **SOA レコード**：ゾーンの管理情報

---
### DNS 委任

委任とは、**親ゾーンが「このサブドメインの管理を、別の権威サーバに任せる」と宣言すること**です。仕組みは**「親ゾーンに、子ゾーンの NS レコードを書く」**ただそれだけ
#### 委任の連鎖を実際にたどる

`web.tokyo.example.com` を解決する流れで見ます。

```
[リゾルバ]
   │  ① 「web.tokyo.example.com の A は？」
   ▼
[ルート(.) 権威サーバ]
   │  ② 「.com の NS はこれ」← ルートが .com に委任している
   ▼
[.com 権威サーバ]
   │  ③ 「example.com の NS はこれ」← .com が example.com に委任
   ▼
[example.com 権威サーバ]
   │  ④ 「tokyo.example.com の NS はこれ」← ここで tokyo に委任
   │      （★ example.com ゾーン内に tokyo の NS レコードがある）
   ▼
[tokyo.example.com 権威サーバ]  ← Route 53 のホストゾーン
   │  ⑤ 「web.tokyo.example.com の A は 192.0.2.10」← 確定回答
   ▼
[リゾルバ] → 192.0.2.10 へ接続
```

各段の「NS はこれ」という案内が委任です。委任が連鎖してルートから末端までつながって初めて、世界中から名前解決できます。

#### 委任が切れると起きること（NS Glue / Lame Delegation）

委任で最も多い事故は、**親に NS を書き忘れる / 間違える** ことです。

`tokyo.example.com` をホストゾーンとして新規作成すると、Route 53 は `tokyo` ゾーン内に NS（①）を自動で作りますが、**親 `example.com` ゾーンへの NS（②）は自動では作られません**。手動で `example.com` ホストゾーンに `tokyo.example.com` の NS レコードを追加する必要があります。

```
やるべきこと:
  example.com ホストゾーン に追加
    NS  tokyo.example.com  →  ns-1.awsdns-01.com.
                                ns-2.awsdns-02.net.   （tokyo ゾーンのNS 4本をコピー）
```

これを忘れると、外からは `example.com` までしかたどれず `tokyo.*` が一切引けない、という「委任切れ（Lame Delegation）」状態になります。前回お話しした「ホストゾーンを分けたら委任が必要」というのは、まさにこの②のことです。

### DNS を Route 53 がどうラップしているか

Route 53 は DNS を置き換える独自技術ではなく、**標準的な DNS プロトコルの「権威サーバ」を AWS がマネージドで提供しているだけ**で

| DNS の概念         | Route 53 での対応物                             |
| --------------- | ------------------------------------------ |
| 権威ネームサーバ        | Route 53 のインフラ（4台のNS）                      |
| ゾーンファイル         | ホストゾーン                                     |
| ゾーンの委任          | 親ゾーンへの NS レコード登録                           |
| A/CNAME/MX レコード | コンソール/API で作るレコード                          |
| （Route 53 独自）   | **Alias レコード**、ヘルスチェック、加重/レイテンシ/位置情報ルーティング |

### FQDN（完全修飾ドメイン名）

先に FQDN を押さえておくと、後の説明が分かりやすくなります。

FQDN（Fully Qualified Domain Name）は、**ルートから下までを省略なく完全に書いたドメイン名**です。実は末尾に「ルートを表すドット」が付くのが正式な姿です。

```
web . tokyo . example . com .
 │      │       │        │   └─ ルート (root)     ← 通常は省略される
 │      │       │        └───── TLD (トップレベルドメイン)
 │      │       └────────────── 2nd レベルドメイン
 │      └────────────────────── サブドメイン
 └───────────────────────────── ホスト名
```

階層は**右から左**に向かって具体的になります。読み解くときは右端のルートから「.com を管理する人 → その中の example → その中の tokyo → その中の web」とたどります。

- `web` だけ → ホスト名（修飾されていない、文脈依存）
- `web.tokyo.example.com.` → FQDN（末尾ドット付き、世界で一意に確定）

ゾーンファイルでは、この**末尾ドットの有無が決定的に重要**
